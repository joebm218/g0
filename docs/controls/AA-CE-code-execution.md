## Overview

AI agents that generate and execute code introduce a uniquely dangerous attack surface. Unlike traditional software where code is written by developers and reviewed before deployment, agent-generated code is created at runtime in response to user requests, model reasoning, or tool invocations. This code may be executed in environments with varying degrees of isolation, ranging from fully sandboxed containers to the agent's own process space.

The AA-CE domain addresses risks across five sub-categories: code generation safety, detection of unsafe code patterns, sandbox escape vectors, hallucinated dependency references (slopsquatting), and framework-specific execution risks. Together, these 80 controls provide comprehensive coverage of the code execution attack surface in agentic AI systems.

### Threat Model

An attacker may exploit code execution in the following ways:

- **Direct prompt injection**: Crafting user inputs that cause the agent to generate and execute malicious code.
- **Indirect prompt injection**: Planting malicious instructions in documents, tool outputs, or memory that later influence code generation.
- **Sandbox escape**: Exploiting weaknesses in the execution sandbox to gain access to the host system.
- **Supply chain poisoning**: Causing the agent to install or reference malicious or non-existent packages (slopsquatting).
- **Lateral movement**: Using code execution capabilities to access credentials, network resources, or other agents.

### Applicability

These controls apply to any agentic system that:

- Generates code (in any language) as part of its operation
- Executes generated code in any environment
- Invokes shell commands, database queries, or system calls
- Uses code execution tools provided by agent frameworks
- Installs or references software packages as part of code generation

## Sub-Category 1: Code Generation Safety

**Control Range**: AA-CE-001 through AA-CE-020
**Mode**: Both (Static + Dynamic)
**Rationale**: Code generated by AI agents at runtime bypasses traditional code review processes. Without proper constraints, generated code can perform arbitrary operations on the host system, access sensitive data, and create persistent backdoors.

### AA-CE-001: Unsandboxed Python Code Execution

| Field              | Value                                                        |
|--------------------|--------------------------------------------------------------|
| **Control ID**     | AA-CE-001                                                    |
| **Severity**       | CRITICAL                                                     |
| **Mode**           | Both                                                         |
| **OWASP**         | ASI05                                                        |

**Description**: The agent generates Python code and executes it directly in the host process or an unsandboxed environment. This grants generated code full access to the operating system, filesystem, network, and any credentials available to the agent process.

**Rationale**: Python's `exec()` and `eval()` functions, when used to run agent-generated code without sandboxing, provide the equivalent of arbitrary code execution. An attacker who can influence code generation (via prompt injection or poisoned context) can achieve full system compromise.

**Test Procedure**:
1. Prompt the agent to generate and execute Python code.
2. Verify whether the code runs in a sandboxed environment (container, VM, restricted interpreter) or directly in the agent's process.
3. Attempt to access system resources (filesystem, network, environment variables) from generated code.
4. Verify that process isolation exists between the agent and the code execution environment.

**Remediation**:
- Execute all generated Python code in an isolated sandbox (e.g., Docker container, gVisor, Firecracker VM, or a restricted Python interpreter such as RestrictedPython).
- Apply seccomp profiles to limit system calls available to the sandbox.
- Use namespaces (PID, network, mount) to isolate the execution environment.
- Never execute generated code in the same process as the agent.

### AA-CE-002: Unsandboxed Shell Command Execution

| Field              | Value                                                        |
|--------------------|--------------------------------------------------------------|
| **Control ID**     | AA-CE-002                                                    |
| **Severity**       | CRITICAL                                                     |
| **Mode**           | Both                                                         |
| **OWASP**         | ASI05                                                        |

**Description**: The agent generates shell commands (bash, sh, zsh, PowerShell) and executes them directly on the host system without restriction or sandboxing.

**Rationale**: Shell commands have unrestricted access to the operating system. An attacker who can influence the agent to generate malicious shell commands can exfiltrate data, install backdoors, modify system configuration, or pivot to other systems on the network.

**Test Procedure**:
1. Prompt the agent to generate and execute shell commands.
2. Attempt to execute commands that read sensitive files (e.g., `/etc/shadow`), modify system configuration, or establish network connections.
3. Verify whether a command whitelist or sandbox restricts which commands can be executed.
4. Test command chaining (`;`, `&&`, `||`, pipes) and command substitution (`$(...)`, backticks).

**Remediation**:
- Maintain an explicit whitelist of allowed commands.
- Execute commands in an isolated container or VM.
- Parse and validate commands before execution; reject commands with chaining, substitution, or redirection operators unless explicitly allowed.
- Use a restricted shell (rbash) or command wrapper that enforces the whitelist.
- Log all executed commands with full arguments for audit.

### AA-CE-003: Direct SQL Execution Against Production Database

| Field              | Value                                                        |
|--------------------|--------------------------------------------------------------|
| **Control ID**     | AA-CE-003                                                    |
| **Severity**       | CRITICAL                                                     |
| **Mode**           | Both                                                         |
| **OWASP**         | ASI05                                                        |

**Description**: The agent generates SQL queries and executes them directly against a production database without parameterization, validation, or use of read-only connections.

**Rationale**: Direct SQL execution against production databases can lead to data exfiltration, data modification, data deletion, or denial of service. Agent-generated SQL is particularly dangerous because the model may be influenced by adversarial inputs to generate destructive queries (DROP TABLE, DELETE, UPDATE without WHERE).

**Test Procedure**:
1. Prompt the agent to query a database and observe the generated SQL.
2. Verify whether parameterized queries are used or if string concatenation builds the query.
3. Attempt to generate DDL statements (CREATE, ALTER, DROP) and DML modification statements (INSERT, UPDATE, DELETE).
4. Verify whether the database connection is read-only.
5. Test SQL injection through user-provided values in the prompt.

**Remediation**:
- Use read-only database connections for agent-generated queries.
- Enforce parameterized queries exclusively; never concatenate user input into SQL strings.
- Maintain a whitelist of allowed tables and columns.
- Implement query complexity limits (row count, execution time, joins).
- Use a database proxy that enforces query policies.
- Run queries against a replica, never against the primary production database.

### AA-CE-004: Generated Code Has Network Access

| Field              | Value                                                        |
|--------------------|--------------------------------------------------------------|
| **Control ID**     | AA-CE-004                                                    |
| **Severity**       | CRITICAL                                                     |
| **Mode**           | Both                                                         |
| **OWASP**         | ASI05                                                        |

**Description**: Code generated by the agent can make outbound network connections, enabling data exfiltration, communication with command-and-control servers, or attacks on internal network services.

**Rationale**: Network access from generated code is the primary vector for data exfiltration. An attacker who achieves code generation influence can exfiltrate sensitive data, download additional payloads, or scan internal networks that are not accessible from external positions.

**Test Procedure**:
1. Generate code that attempts HTTP/HTTPS requests to external endpoints.
2. Generate code that attempts DNS lookups.
3. Generate code that attempts to connect to internal network services.
4. Verify whether network namespace isolation or firewall rules restrict outbound connections.

**Remediation**:
- Execute generated code in a network-isolated environment (no network namespace, or strict egress firewall rules).
- If network access is required, use an allowlist of permitted domains/IPs.
- Implement a network proxy that inspects and logs all outbound traffic from the sandbox.
- Block DNS resolution from the sandbox to prevent DNS-based exfiltration.

### AA-CE-005: Generated Code Has Broad Filesystem Access

| Field              | Value                                                        |
|--------------------|--------------------------------------------------------------|
| **Control ID**     | AA-CE-005                                                    |
| **Severity**       | CRITICAL                                                     |
| **Mode**           | Both                                                         |
| **OWASP**         | ASI05                                                        |

**Description**: Code generated by the agent can access filesystem paths outside of a designated sandbox directory, including system files, configuration files, other users' data, and agent internals.

**Rationale**: Unrestricted filesystem access enables reading credentials, configuration files, and sensitive data. It also enables writing malicious files (e.g., cron jobs, SSH keys, startup scripts) that persist beyond the code execution session.

**Test Procedure**:
1. Generate code that attempts to read files outside the sandbox (e.g., `/etc/passwd`, `~/.ssh/id_rsa`, agent configuration files).
2. Generate code that attempts to write files outside the sandbox.
3. Generate code that traverses directories using `../` sequences.
4. Verify that filesystem mounts are restricted to the sandbox directory.

**Remediation**:
- Mount only a dedicated sandbox directory into the execution environment.
- Use read-only mounts for any files that the generated code needs to read but not modify.
- Apply filesystem namespacing (mount namespace) to restrict the visible filesystem.
- Implement path canonicalization and validation to prevent directory traversal.

### AA-CE-006: Generated Code Can Install Packages

| Field              | Value                                                        |
|--------------------|--------------------------------------------------------------|
| **Control ID**     | AA-CE-006                                                    |
| **Severity**       | CRITICAL                                                     |
| **Mode**           | Both                                                         |
| **OWASP**         | ASI05                                                        |

**Description**: Agent-generated code can invoke package managers (pip, npm, apt, brew, cargo, etc.) to install software packages. This can introduce malicious or vulnerable dependencies into the execution environment.

**Rationale**: Package installation from generated code creates a supply chain attack vector. The agent may be manipulated into installing typosquatted packages, packages with post-install scripts that execute malicious code, or packages with known vulnerabilities. Installed packages persist in the environment and may affect subsequent executions.

**Test Procedure**:
1. Generate code that includes `pip install`, `npm install`, or equivalent commands.
2. Verify whether package managers are available in the execution environment.
3. Test installation of packages with known malicious post-install scripts (in a safe test environment).
4. Verify whether a package allowlist restricts installable packages.

**Remediation**:
- Pre-install all required packages in the sandbox image; remove package managers from the runtime environment.
- If dynamic installation is required, maintain an explicit allowlist of permitted packages and versions.
- Use a private package registry that mirrors only vetted packages.
- Verify package checksums before installation.
- Block network access to public package registries from the sandbox.

### AA-CE-007: Generated Code Can Access Environment Variables

| Field              | Value                                                        |
|--------------------|--------------------------------------------------------------|
| **Control ID**     | AA-CE-007                                                    |
| **Severity**       | CRITICAL                                                     |
| **Mode**           | Both                                                         |
| **OWASP**         | ASI05                                                        |

**Description**: Code generated by the agent can read environment variables from the execution environment, which may contain API keys, database credentials, cloud provider tokens, and other secrets.

**Rationale**: Environment variables are a common mechanism for passing secrets to applications. If generated code can access these variables, it can extract credentials and either use them directly or exfiltrate them to an attacker-controlled endpoint.

**Test Procedure**:
1. Generate code that reads `os.environ` (Python), `process.env` (Node.js), or equivalent.
2. Verify whether the execution environment inherits the agent's environment variables.
3. Check for specific high-value variables (AWS_ACCESS_KEY_ID, DATABASE_URL, API_KEY, etc.).
4. Test whether environment variables are cleared or isolated in the sandbox.

**Remediation**:
- Clear all environment variables in the sandbox except those explicitly required for code execution.
- Never pass the agent's own secrets to the code execution environment.
- Use a secrets manager with scoped, short-lived tokens instead of environment variables.
- Audit and log any environment variable access from generated code.

### AA-CE-008: Generated Code Can Spawn Child Processes

| Field              | Value                                                        |
|--------------------|--------------------------------------------------------------|
| **Control ID**     | AA-CE-008                                                    |
| **Severity**       | CRITICAL                                                     |
| **Mode**           | Both                                                         |
| **OWASP**         | ASI05                                                        |

**Description**: Code generated by the agent can create child processes, enabling it to run arbitrary executables, shell commands, or scripts that may not be subject to the same restrictions as the parent code execution.

**Rationale**: Process spawning is a common sandbox escape technique. Generated code may use child processes to execute system commands, run additional scripts, or create background processes that persist after the main execution completes. Child processes may inherit elevated privileges or access to resources not available to the sandboxed code.

**Test Procedure**:
1. Generate code that uses `subprocess.Popen()`, `os.system()`, `child_process.exec()`, or equivalent.
2. Verify whether the sandbox restricts process creation (e.g., via seccomp, PID namespace limits).
3. Test whether child processes can access resources not available to the parent process.
4. Test creation of background/daemon processes that survive the main execution.

**Remediation**:
- Use seccomp profiles to block fork/exec system calls unless explicitly required.
- Limit the number of processes via PID namespace or cgroup limits.
- Ensure child processes inherit the same restrictions as the parent.
- Monitor and terminate orphaned child processes after the main execution completes.

### AA-CE-009: Generated Code Has No Execution Timeout

| Field              | Value                                                        |
|--------------------|--------------------------------------------------------------|
| **Control ID**     | AA-CE-009                                                    |
| **Severity**       | CRITICAL                                                     |
| **Mode**           | Both                                                         |
| **OWASP**         | ASI05                                                        |

**Description**: Code generated by the agent can execute indefinitely without a timeout, enabling denial of service through infinite loops, long-running computations, or resource exhaustion.

**Rationale**: Without execution timeouts, a single piece of generated code (whether malicious or accidentally flawed) can consume compute resources indefinitely, blocking other users, increasing costs, and potentially crashing the system.

**Test Procedure**:
1. Generate code containing an infinite loop (`while True: pass`).
2. Generate code with exponential-time computation.
3. Measure whether the execution is terminated after a reasonable timeout.
4. Verify that the timeout applies to both CPU time and wall-clock time.

**Remediation**:
- Enforce a wall-clock timeout for all code executions (e.g., 30 seconds for interactive, 5 minutes for batch).
- Enforce CPU time limits via cgroup or ulimit.
- Monitor execution duration and alert on abnormal patterns.
- Implement graceful termination (SIGTERM followed by SIGKILL) on timeout.

### AA-CE-010: Generated Code Output Not Size-Limited

| Field              | Value                                                        |
|--------------------|--------------------------------------------------------------|
| **Control ID**     | AA-CE-010                                                    |
| **Severity**       | CRITICAL                                                     |
| **Mode**           | Both                                                         |
| **OWASP**         | ASI05                                                        |

**Description**: Output from generated code execution (stdout, stderr, return values, generated files) is not size-limited, enabling memory exhaustion, log flooding, or context window overflow in the agent.

**Rationale**: Unlimited output can exhaust memory in the agent process, overwhelm logging infrastructure, or flood the agent's context window with irrelevant data that displaces important instructions. An attacker can use this to create denial of service or to manipulate the agent's context.

**Test Procedure**:
1. Generate code that produces very large output (e.g., `print("A" * 10**9)`).
2. Generate code that writes very large files.
3. Verify whether output is truncated before being returned to the agent.
4. Test whether output size limits apply to both stdout and stderr.

**Remediation**:
- Truncate stdout/stderr output to a configured maximum size (e.g., 64KB).
- Limit generated file sizes via filesystem quotas.
- Implement streaming output processing with size checks.
- Return a summary/truncation notice to the agent when output is limited.

| Field              | Value                                                        |
|--------------------|--------------------------------------------------------------|
| **Control ID**     | AA-CE-011                                                    |
| **Severity**       | HIGH                                                         |
| **Mode**           | Both                                                         |
| **OWASP**         | ASI05                                                        |

**Description**: Code generated by the agent can access the agent's own API keys, authentication tokens, or service account credentials, enabling credential theft and impersonation.

**Rationale**: If generated code runs in the same process or has access to the same credential stores as the agent, it can extract credentials and use them for unauthorized actions or exfiltrate them to an attacker.

**Test Procedure**:

### AA-CE-020: Multi-Language Sandbox Bypass

| Field              | Value                                                        |
|--------------------|--------------------------------------------------------------|
| **Control ID**     | AA-CE-020                                                    |
| **Severity**       | HIGH                                                         |
| **Mode**           | Both                                                         |
| **OWASP**         | ASI05                                                        |

**Description**: The agent can generate code in multiple programming languages, and an attacker can request a language that bypasses language-specific sandbox restrictions.

## Sub-Category 2: Eval/Exec Detection

**Control Range**: AA-CE-021 through AA-CE-040
**Mode**: Static
**Tier**: Stable
**Rationale**: Many code execution vulnerabilities stem from unsafe patterns in the agent's own source code. Static analysis can detect these patterns before deployment, preventing them from reaching production.

### AA-CE-021: Python eval() in Agent Code

| Field              | Value                                                        |
|--------------------|--------------------------------------------------------------|
| **Control ID**     | AA-CE-021                                                    |
| **Severity**       | CRITICAL                                                     |
| **Mode**           | Static                                                       |
| **Tier**           | Stable                                                       |

**Description**: The agent's source code contains calls to Python's `eval()` function, which evaluates arbitrary Python expressions from a string argument.

**Rationale**: `eval()` converts string input into executable code. If any part of the evaluated string is derived from user input, model output, or tool results, it creates an arbitrary code execution vulnerability. Even seemingly safe uses of `eval()` can be exploited through creative expression construction.

**Test Procedure**:
1. Scan agent source code for all instances of `eval(`.
2. Trace the data flow to determine whether the argument is ever derived from untrusted input.
3. Check for obfuscated variants (e.g., `getattr(builtins, 'eval')`).

**Remediation**:
- Replace `eval()` with `ast.literal_eval()` for parsing Python literals.
- Use `json.loads()` for parsing JSON strings.
- Implement dedicated parsers for specific expression formats.
- If `eval()` is absolutely required, use a restricted global/local namespace.
- Flag all `eval()` usage in code review and require explicit security justification.

### AA-CE-079: Custom Code Execution Endpoint Without Authentication

| Field              | Value                                                        |
|--------------------|--------------------------------------------------------------|

| **Severity**       | HIGH                                                         |
| **Mode**           | Static                                                       |
| **Tier**           | Stable                                                       |

**Description**: A custom API endpoint used by the agent for code execution does not require authentication, allowing anyone who can reach the endpoint to execute arbitrary code.

**Rationale**: Unauthenticated code execution endpoints are equivalent to an open remote code execution vulnerability. If the endpoint is accessible from the network, any attacker who discovers it can execute arbitrary code on the server.

### AA-CE-080: Agent Framework Auto-Executes Code in Model Responses

| Field              | Value                                                        |
|--------------------|--------------------------------------------------------------|
| **Control ID**     | AA-CE-080                                                    |
| **Severity**       | HIGH                                                         |
| **Mode**           | Static                                                       |
| **Tier**           | Stable                                                       |

**Description**: The agent framework automatically detects and executes code blocks in model responses without explicit tool invocation, human approval, or safety checks.

**Rationale**: Automatic code execution from model responses removes all control points between code generation and execution. If the model is influenced by prompt injection, the injected code executes immediately without any opportunity for validation.

**Test Procedure**:
1. Verify whether the agent framework parses model responses for code blocks.
2. Test whether code blocks in model responses are automatically executed.
3. Verify whether execution requires explicit tool calls or function calls.

**Remediation**:
- Require explicit tool/function call mechanisms for code execution (not text-based detection).
- Implement human-in-the-loop confirmation for code execution.
- If auto-execution is required, apply all standard sandboxing and validation controls.
- Log all auto-executed code with the model response that triggered it.
